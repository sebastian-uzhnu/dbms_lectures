# MySQL: Все про JOIN (Об'єднання Таблиць)

Сьогодні ми перестанемо гратися в "пісочниці" і розберемо одну з найважливіших тем у реляційних базах даних. Якщо ви зрозумієте `JOIN` — ви зрозумієте SQL.

Також ми зачепимо тему продуктивності і типових помилок.

---

## Підготовка: Створюємо таблиці

Перш ніж щось об'єднувати, треба це створити.
У реляційних базах ми розбиваємо дані на логічні частини: клієнти окремо, замовлення окремо.

> **Важливий момент:** `JOIN` працює, навіть якщо у вас немає офіційного зв'язку (Foreign Key). Ви можете об'єднати будь-які дві таблиці, якщо в них є логічно пов'язані дані (наприклад, `user_id`). Але **Foreign Key** допомагає тримати порядок (не дає видалити юзера, якщо у нього є замовлення).

Давайте створимо дві таблиці: `users` та `orders`.

### 1. Зв'язок на "Чесному слові" (Без FK)

Ви можете просто створити колонку `user_id` і записувати туди числа. Це працюватиме для `JOIN`, але це небезпечно.

```sql
CREATE TABLE users_simple (id INT, name VARCHAR(100));
CREATE TABLE orders_simple (id INT, user_id INT, amount INT);
-- Ніхто не заборонить вам записати сюди user_id = 99999, якого не існує.
```

### 2. Офіційний зв'язок (Foreign Key)

Це "суворий поліцейський" у базі даних.

**Foreign Key (Зовнішній Ключ)** — це правило, яке каже: "У колонку `user_id` таблиці `orders` можна записати ТІЛЬКИ ті значення, які вже існують в колонці `id` таблиці `users`".

Якщо ви спробуєте видалити користувача, у якого є замовлення — база скаже "Стоп! Спочатку видали замовлення". Це називається **Referential Integrity** (Цілісність даних).

```sql
-- 1. Створюємо таблицю користувачів
CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100),
    email VARCHAR(100)
);

-- 2. Створюємо таблицю замовлень
CREATE TABLE orders (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT,
    amount DECIMAL(10, 2),
    status VARCHAR(20),
    -- Ось тут ми створюємо "офіційний" зв'язок:
    CONSTRAINT fk_user FOREIGN KEY (user_id) REFERENCES users(id)
    ON DELETE CASCADE -- Якщо видалити юзера, видаляться і його замовлення
);
```

**Наповнимо їх даними:**

```sql
INSERT INTO users (name, email) VALUES
('Оля', 'olya@mail.com'),
('Петро', 'petro@mail.com'),
('Іван', 'ivan@mail.com'),
('Марія', 'maria@mail.com');

INSERT INTO orders (user_id, amount, status) VALUES
(1, 200.00, 'compl'),  -- Оля купила щось за 200
(1, 500.00, 'pend'),   -- Оля купила щось за 500
(2, 300.00, 'compl');  -- Петро купив щось за 300
-- Іван та Марія нічого не купували!
```

**Що ми маємо зараз:**

**Таблиця `users` (Клієнти):**
| id | name |
|:---:|---|
| 1 | **Оля** |
| 2 | **Петро**|
| 3 | **Іван** |
| 4 | **Марія**|

**Таблиця `orders` (Замовлення):**
| id | user_id | amount |
|:---:|:---:|---|
| 101 | 1 | 200 |
| 102 | 1 | 500 |
| 103 | 2 | 300 |

## 1. Що таке JOIN і для чого він?

**JOIN** (Об'єднання) — це операція, яка дозволяє нам тимчасові "зібрати" ці розкидані дані в одну віртуальну таблицю для звіту.

### Як це працює?

У нас є "клеми", якими ми з'єднуємо таблиці.

- **Primary Key (PK):** Унікальний ID запису (наприклад, `users.id`).
- **Foreign Key (FK):** Посилання на ID в іншій таблиці (наприклад, `orders.user_id`).

> **Різниця між ON та WHERE:**
>
> - `ON` визначає **як** ми з'єднуємо таблиці (за яким полем).
> - `WHERE` фільтрує вже **з'єднаний** результат (кого залишити).

---

## 2. INNER JOIN (Внутрішнє об'єднання)

> _"Покажи мені тільки ідеальні пари"_

Це найсуворіший тип. Якщо запис в одній таблиці не має пари в іншій — він вилітає.
Це як перетин двох кіл на діаграмі Венна.

```sql
SELECT users.name, orders.amount
FROM users
INNER JOIN orders ON users.id = orders.user_id;
```

**Результат:**
| name | amount |
|---|---|
| Оля | 200 |
| Оля | 500 |
| Петро | 300 |

- **Іван та Марія** зникли (у них немає замовлень).

---

## 3. LEFT JOIN (Ліве об'єднання)

> _"Клієнт завжди правий (навіть якщо нічого не купив)"_

Ми беремо **всі** рядки з лівої таблиці (`users`). Якщо для них знайшлися замовлення — супер, показуємо. Якщо ні — заповнюємо пустотою (`NULL`).

**Секрет:** Це найчастіший тип JOIN-у в реальному житті.

```sql
SELECT users.name, orders.amount
FROM users
LEFT JOIN orders ON users.id = orders.user_id;
```

**Результат:**
| name | amount |
|---|---|
| Оля | 200 |
| Петро | 300 |
| **Іван** | **NULL** |
| **Марія** | **NULL** |

- **Іван та Марія** на місці! Ми бачимо, що вони є в базі, просто `amount` у них `NULL`.

### Лайфхак: Як знайти тих, хто нічого не купив?

Це класичне питання на співбесіді.

```sql
SELECT users.name
FROM users
LEFT JOIN orders ON users.id = orders.user_id
WHERE orders.id IS NULL; -- Шукаємо пустоту
```

---

## 4. RIGHT JOIN (Праве об'єднання)

Це дзеркальне відображення `LEFT JOIN`. Беремо все з правої таблиці (`orders`), навіть якщо немає власника.

> ⚠️ **Порада:** У 99% випадків краще використовувати `LEFT JOIN`. Читати зліва направо зручніше для нашого мозку. `RIGHT JOIN` часто просто перевертають і роблять `LEFT JOIN`.

---

## 5. CROSS JOIN (Декартовий добуток)

> _"Хаос. Кожен з кожним."_

Якщо ми не вкажемо умову `ON`, база просто поєднає кожен рядок з першої таблиці з кожним рядком другої.
Якщо у вас 100 юзерів і 100 замовлень, ви отримаєте **10 000** рядків (100 × 100).

```sql
SELECT * FROM users CROSS JOIN orders;
```

**Небезпека:** Якщо таблиці великі, ви можете "покласти" сервер, згенерувавши мільярд рядків сміття. Використовуйте обережно (наприклад, для генерації комбінацій розмірів і кольорів).

---

## 6. Порядок виконання запиту

Це **критично важливо** розуміти. SQL виконується не так, як ви його пишете.

**Як ви пишете:**
`SELECT` → `FROM` → `JOIN` → `WHERE`...

**Як база це виконує:**

1.  **FROM / JOIN** (Збираємо дані до купи)
2.  **WHERE** (Фільтруємо сирі дані)
3.  **GROUP BY** (Групуємо)
4.  **HAVING** (Фільтруємо групи)
5.  **SELECT** (Вибираємо колонки, які треба показати)
6.  **ORDER BY** (Сортуємо результат)

> **Чому це важливо?**
> Тому що, якщо ви зробите `LEFT JOIN`, а потім у `WHERE` напишете умову по правій таблиці (наприклад, `WHERE orders.amount > 100`), ваш `LEFT JOIN` перетвориться на `INNER JOIN`! Бо `NULL` не може бути більше 100, і ці рядки відфільтруються.

## 7. Типові помилки студентів

1.  **Забули `ON`:** Отримали CROSS JOIN і сервер "завис".
2.  **`WHERE` вбиває `LEFT JOIN`:** Див. пункт вище. Якщо хочете фільтрувати праву таблицю при LEFT JOIN, пишіть умову прямо в `ON`!
    - `WHERE orders.status = 'active'` (відфільтрує NULL-и)
    - `ON ... AND orders.status = 'active'` (залишить NULL-и)
3.  **Дублікати:** Якщо один юзер зробив 5 замовлень, `JOIN` розмножить цього юзера 5 разів. Якщо ви рахуєте кількість юзерів — використовуйте `COUNT(DISTINCT users.id)`.

## 8. Продуктивність та EXPLAIN

"Чому мій запит виконується 10 секунд?"

### Nested Loop vs Hash Join

Уявіть, як база шукає відповідності:

- **Nested Loop:** Бере кожного юзера і біжить перевіряти весь список замовлень. Повільно.
- **Hash Join:** Будує хеш-таблицю (словник) і миттєво знаходить пари. Швидко.

### Індекси

Кожне поле, яке використовується в `ON` (наприклад, `user_id`), **обов'язково** має бути проіндексоване! Без індексу база буде перечитувати всю таблицю для кожного рядка.

### Команда EXPLAIN

Перед будь-яким складним запитом пишіть `EXPLAIN`.

```sql
EXPLAIN SELECT * FROM users JOIN orders ...
```

Вона покаже, чи використовує база індекси (дивіться колонку `key`) і скільки рядків їй довелося перебрати.

### Задача 2: Сума покупок кожного

Виведіть ім'я та загальну суму витрачених грошей. Якщо не купував — 0.

<details>
<summary>Показати відповідь</summary>

```sql
SELECT users.name, COALESCE(SUM(orders.amount), 0) as total
FROM users
LEFT JOIN orders ON users.id = orders.user_id
GROUP BY users.id, users.name;
```

</details>

### Задача 3: Знайти замовлення-"сироти"

Знайдіть id замовлень, які посилаються на неіснуючих юзерів.

<details>
<summary>Показати відповідь</summary>

```sql
SELECT orders.id
FROM orders
LEFT JOIN users ON orders.user_id = users.id
WHERE users.id IS NULL;
```

</details>
